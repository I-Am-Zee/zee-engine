## FINDING 1: The Image.astro Primitive — How It Actually Routes

`Image.astro` has smart routing logic (lines 42-58):

- If src starts with `/images/`, it STRIPS the `/images/` prefix and prepends `/${brandId}/`.
- It then builds the full URL using `PUBLIC_IMAGE_GATEWAY_URL`.
- In DEV mode (no gateway), it falls back to the local `/images/` folder.

Example transformation:
  - Input:  /images/products/gold-ring.jpg
  - Output: https://assets.zeliavance.com/zelia-vance/products/gold-ring.jpg

This is working correctly and is already multi-tenant aware.
HOWEVER: This ONLY works for paths going through `Image.astro`.
Plain `<img>` tags (like in snipcart-templates.html) are NOT processed by this.

## FINDING 4: The Snipcart Cart Summary — Image Routing Status

File: public/snipcart-templates.html

Snipcart renders its own cart sidebar using its own Vue-based templates.
The cart thumbnail image is rendered by Snipcart's own JS from the `data-item-image`
attribute on your Add To Cart buttons in product YAML.

Snipcart does NOT go through `Image.astro`. It renders the image URL directly.
Whatever string is in `data-item-image` is what appears in the cart sidebar thumbnail.

If that string is `/images/products/...`, Snipcart will request it from your Astro server,
which serves local files in dev. In production (Cloudflare Pages), there are no local files —
so the cart thumbnails will be BROKEN unless the `data-item-image` value is a full absolute URL.

🔴 ACTION NEEDED: Product YAML `image` fields that are used for Snipcart's
`data-item-image` must be FULL absolute R2 URLs (e.g., https://assets.zeliavance.com/...)
in production, not relative `/images/...` paths.

This means the image field in your product YAML must carry the full URL.
`Image.astro` can handle both — but Snipcart cannot.
