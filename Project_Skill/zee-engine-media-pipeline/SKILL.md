# Zee Engine — Media & Image Pipeline Specialist

> **Prerequisite:** Read `zee-engine-core` skill first.
> **Always read before:** Touching `Image.astro`, `images.ts`, any component that displays a product/lookbook image, or any Alpine component with `:src` bindings.

---

## 1. The Three Actors — Code, Context, Browser

When using the `Image.astro` primitive, there are **three distinct actors**. Confusing them is the root cause of all image bugs.

| Actor | What It Is | What It Does |
|---|---|---|
| **The Menu (`srcset`)** | A list of available image URLs at different widths | Generated by `Image.astro` via the `resolveSrcset()` utility and the R2 Worker |
| **The Context (`sizes`)** | A CSS media query string you provide | Tells the browser how large the image will appear on screen *before* any CSS loads |
| **The Decision (The Browser)** | The browser runtime | Reads `sizes`, calculates viewport + screen density, consults `srcset`, and downloads the mathematically optimal image autonomously |

> **The code does not choose which image downloads. The browser does.**
> Your responsibility is to provide an accurate `sizes` value. If you lie about the size, the browser will download the wrong image.

---

## 2. The Image Utility — `scripts/utils/images.ts`

Two pure functions power the entire server-side pipeline. Never reinvent this logic.

### `resolveImage(path, width?)`

Transforms a local-style path into a fully-qualified R2 Gateway URL.

```typescript
// Input:  "/images/products/ring.webp"
// Output: "https://assets.zeliavance.com/zelia-vance/products/ring.webp"
// With width: "https://assets.zeliavance.com/zelia-vance/products/ring.webp?w=800"

// Ghost Operator path transformation rule:
//   /images/products/ring.webp  →  /{brandId}/products/ring.webp  (R2 prefix)
//   /anything-else              →  /{brandId}/anything-else
//   http://...                  →  unchanged (just appends ?w= if width provided)
```

**Key behaviours:**
- Uses `PUBLIC_IMAGE_GATEWAY_URL` from env. If empty (dev without gateway), the URL has no host prefix → falls back to local junction files.
- Uses `PUBLIC_BRAND_ID` to inject the brand namespace into the R2 path, ensuring brand isolation at the asset level.
- Works in both Astro (build-time) and Alpine (client-side) contexts — it is deliberately written as isomorphic JS.

### `resolveSrcset(path, widths?)`

Calls `resolveImage` for each width and joins as a valid `srcset` string.

```typescript
resolveSrcset("/images/products/ring.webp", [400, 800, 1200])
// → "https://assets.../zelia-vance/products/ring.webp?w=400 400w,
//    https://assets.../zelia-vance/products/ring.webp?w=800 800w,
//    https://assets.../zelia-vance/products/ring.webp?w=1200 1200w"
```

Default widths if unspecified: `[400, 800, 1200]`.

## 3. Standard Image Usage — Prop API

```astro
<Image
  src="/images/products/ring.webp"   -- REQUIRED. Local-style path or http URL.
  alt="Diamond ring close-up"        -- REQUIRED. Non-empty for accessibility.
  ratio="square"     -- Optional. "square"|"portrait"|"landscape"|"auto". Default: "auto"
  widths={[400, 800, 1200]}          -- Optional. Default: [400, 800, 1200]
  sizes="(max-width: 768px) 100vw, 50vw"  -- Optional. Default: "(max-width: 800px) 100vw, 800px"
  loading="lazy"     -- Optional. Use "eager" + fetchpriority="high" for LCP images.
  fade={true}        -- Optional. Opacity fade-in on load. Default: true
/>
```

### Two Rendering Paths

`Image.astro` has two internal branches:

| Path | When | How |
|---|---|---|
| **Remote / Local string** | `src` is a string starting with `/` or `http`, or `:src` is present | Uses a raw `<img>` tag with `resolveImage()` + `resolveSrcset()` for R2 URLs |
| **Astro local import** | `src` is an imported asset object | Uses `<AstroImage>` (Astro's built-in optimizer) for full format conversion |

**Rule:** All product/lookbook images are strings (paths or URLs). They always go through the Remote path → R2 pipeline. Only use imported assets for static design elements like brand logos.

## 4. Responsive Sizes — The Critical Contract

The `sizes` attribute is **not optional for components**. Getting it wrong costs bandwidth.

### Correct Sizes by Context

| Context | Correct `sizes` value |
|---|---|
| Full-width hero | `"100vw"` |
| Product grid card (2 col mobile, 4 col desktop) | `"(max-width: 640px) 50vw, 25vw"` |
| PDP main image (full width mobile, half desktop) | `"(max-width: 768px) 100vw, 50vw"` |
| SideDrawer thumbnail (~80px) | `"80px"` |
| Lookbook carousel (full mobile, 60vw desktop) | `"(max-width: 768px) 100vw, 60vw"` |
| Related products row | `"(max-width: 768px) 50vw, 20vw"` |

### The Thumbnail Trap (⚠️ High-Bandwidth Bug)

```astro
<!-- WRONG: Tiny 80px UI thumbnail using default sizes -->
<Image src={product.image} alt={product.title} />
<!-- Default sizes="(max-width: 800px) 100vw, 800px" →
     Browser downloads 800px image for an 80px element. Massive waste. -->

<!-- CORRECT: Explicit sizes for small UI elements -->
<Image
  src={product.image}
  alt={product.title}
  widths={[200, 400]}
  sizes="80px"
/>
```

**Rule: Any image rendered at a fixed small size (drawers, thumbnails, avatars, nav icons) MUST have explicit `widths` and `sizes` props.**

---

## 5. The 200w Width — Why It Exists

The engine uses **4 standard widths: `[200, 400, 800, 1200]`** as defaults in some components.

> **200w was added explicitly** for high-performance PDP gallery thumbnails (the small image strip below the main PDP image). Without it, the browser would download a 400px image for a ~120px thumbnail on mobile, wasting bandwidth.

**Do not remove 200w from PDP gallery components.** It is intentional.

For general-purpose images where thumbnails are not a concern, `[400, 800, 1200]` is the standard default.

## 6. The Split Brain Problem (Astro vs. Alpine)

This is the most dangerous image bug in the engine. It occurs whenever a component renders images **dynamically via Alpine.js**.

### Why It Happens

```astro
<!-- Static (Astro) — CORRECT -->
<Image src={product.image} alt={product.title} />
<!-- Astro resolves this at build time:
     /images/products/ring.webp → https://assets.zeliavance.com/zelia-vance/products/ring.webp
     srcset is generated perfectly. ✅ -->

<!-- Dynamic (Alpine) — BROKEN without the fix -->
<img :src="item.image" />
<!-- Alpine is a runtime JS library. The server never sees this binding.
     The browser receives the raw string "/images/products/ring.webp".
     Result: 404 in production (no local files), R2 gateway bypassed. ❌ -->
```

### Why It's Dangerous

- In **dev with junction**: The path `/images/products/ring.webp` resolves locally. The bug is hidden.
- In **production**: There are no local files. The raw path 404s. The R2 gateway is bypassed.
- **No build error.** Astro does not know about `:src` — it's runtime JavaScript.

## 7. The `$engine` Magic Property Bridge (The Fix)

`Image.astro` automatically detects when an Alpine `:src` binding is passed and injects the engine resolver.

### How It Works (Implementation)

```astro
---
// In Image.astro frontmatter:
const alpineSrc = rest[':src'] ? `$engine.image(${rest[':src']})` : undefined;
const alpineSrcset = rest[':src'] ? `$engine.srcset(${rest[':src']})` : undefined;
---

<img
  x-bind:src={alpineSrc}
  x-bind:srcset={alpineSrcset}
  ...
/>
```

When `:src="item.image"` is passed as a prop, the primitive outputs:
- `x-bind:src="$engine.image(item.image)"` — resolves the path through `resolveImage()` at runtime
- `x-bind:srcset="$engine.srcset(item.image)"` — generates the full responsive srcset at runtime

`$engine` is an Alpine magic property registered globally. It calls the same `resolveImage` and `resolveSrcset` functions from `images.ts` — the **exact same logic** as the server-side path.

### Usage in Components

```astro
<!-- Correct: Let Image.astro handle the bridge automatically -->
<Image :src="item.image" alt="Product image" sizes="80px" widths={[200, 400]} />

<!-- NEVER do this manually — it defeats the bridge -->
<img :src="`https://assets.zeliavance.com/zelia-vance/${item.image}`" />
<!-- ↑ Hardcoded URL, not multi-tenant, not responsive, will break on brand change -->
```

**Rule: Never manually construct R2 URLs in Alpine expressions. Always pass `:src` to `Image.astro` and let `$engine` handle it.**

## 8. Dev vs. Production Behaviour

| Environment | `PUBLIC_IMAGE_GATEWAY_URL` | Image Source |
|---|---|---|
| **Production** | `https://assets.zeliavance.com` | R2 Worker → full `srcset` at all widths |
| **Dev (with junction + empty gateway)** | `""` | Local junction files via Vite bridge. No `srcset` (no gateway). |
| **Dev (with gateway set)** | `https://assets.zeliavance.com` | R2 Worker, same as production. Requires internet. |
| **Dev (no junction, no gateway)** | `""` | 404 for product images. `onerror` shows placeholder. |

> **Dev default:** Set `PUBLIC_IMAGE_GATEWAY_URL=` (empty) in `.env` to use local junction files. This is the normal dev workflow.

## 9. Error Prevention Cheat Sheet

| Bug | Symptom | Cause | Fix |
|---|---|---|---|
| 800px image in 80px slot | High mobile data use | Missing `sizes` on thumbnail | Add `sizes="80px" widths={[200, 400]}` |
| Product images 404 in production | Broken image icon | Alpine `:src` bypassing gateway | Pass `:src` to `Image.astro`, use `$engine` bridge |
| All images 404 in dev | Broken across the board | Missing junction or `server.fs.allow` | Check Ghost Operator setup (see `zee-engine-core` skill) |
| R2 URL showing brand prefix twice | Malformed URL | Manually constructing R2 path | Never hardcode R2 URLs. Use `resolveImage()` or `Image.astro`. |
| `onerror` placeholder everywhere in dev | Placeholder shown | Gateway URL set in dev but no internet | Clear `PUBLIC_IMAGE_GATEWAY_URL` in `.env` for local dev |
| Images perfect in dev, 404 in prod | Works locally only | Local junction serving files the R2 bucket doesn't have | Upload missing media to R2 via Rclone |

## 10. Do Not Touch Without Full Context

- `src/scripts/utils/images.ts` — The isomorphic resolver. Changes here affect both server and client rendering.
- `src/components/primitives/Image.astro` — The rendering bridge. Changes to `alpineSrc`/`alpineSrcset` injection affect all dynamic image components.
- The `$engine` Alpine magic property registration — If removed, all Alpine `:src` bindings silently 404 in production.
- The `onerror` attribute on both `<img>` and `<AstroImage>` branches — This is the client-side safety net. Removing it causes layout breaks when files haven't been synced yet.

## 11. Client-Side Fallback Strategy (The `onerror` Safety Net)

### Why Missing Images Are Expected — Not Bugs

The Ghost Operator architecture intentionally separates **content authoring** from **asset syncing**. A CMS operator can define a valid image path (e.g., `/images/theme/home-hero.jpg`) in Keystatic at any point — even before the physical file has been:

- Copied to the local Windows Junction (`public/images/`)
- Uploaded to the R2 bucket via Rclone

This is **by design**, not an error. It means **a valid, correctly-formatted URL will 404** during the window between "path defined in CMS" and "file synced to storage." The engine must handle this gracefully without breaking the layout.

### The Rule: Output the URL, Let the Browser Try, Catch with `onerror`

> **Never write server-side existence checks.** Do not add `fs.existsSync()` guards, API pre-flight calls, or URL validation logic before rendering an image URL. This breaks the pipeline.

The correct strategy is a strict **client-side only** safety net. Every `<img>` tag rendered by `Image.astro` includes this exact `onerror` attribute on both the remote branch and the AstroImage branch:

```html
onerror="this.onerror=null; this.srcset=''; this.src='https://placehold.co/800x800?text=Placeholder';"
```

**What this does, step by step:**

1. `Image.astro` renders the URL it was given (R2 gateway URL or local path) — no existence check.
2. The browser attempts to fetch the image.
3. **If the file exists** → renders normally. No intervention needed.
4. **If the file 404s** → `onerror` fires instantly in the browser.
5. `this.onerror=null` — removes the handler so it can't fire again (prevents infinite loop if the placeholder itself fails).
6. `this.srcset=''` — clears the srcset so the browser doesn't try fallback widths (all would 404 too).
7. `this.src='https://placehold.co/800x800?text=Placeholder'` — swaps in an 800×800 neutral placeholder square.

### Why This Approach Is Correct

| Approach | Problem |
|---|---|
| Server-side `fs.existsSync()` before render | Breaks on production (no local filesystem). Slows every page build. Doesn't help for R2 URLs. |
| Server-side HTTP HEAD request to R2 | Adds latency to every image render. Fails on cold starts. Requires auth. |
| Omitting the image if path is undefined | Causes layout collapse — empty slots break grid alignment. |
| **`onerror` client-side swap** ✅ | Zero server overhead. Zero layout shift. Instant swap. Works for both local and R2 sources. |

### What Agents Must Never Do

```astro
<!-- WRONG: Server-side existence guard -->
---
import { existsSync } from 'fs';
const imgExists = existsSync(`public${product.image}`);
---
{imgExists && <Image src={product.image} alt={product.title} />}

<!-- Why it's wrong:
     - existsSync only works on the local junction, not R2
     - Returns false in production (no junction)
     - Hides content that should show a placeholder instead
     - Zero benefit over the onerror approach -->

<!-- CORRECT: Trust the pipeline, let onerror handle it -->
<Image src={product.image} alt={product.title} />
```

### The Content Entry Workflow This Enables

```
1. CMS operator adds a new product with image path "/images/products/new-ring.webp"
2. Page is saved → path is valid in YAML
3. Engineer syncs media later with Rclone
4. Between step 2 and 3:
   - The product page RENDERS (no crash, no server error)
   - The image slot shows the neutral 800×800 placeholder
   - Layout is intact — grid, spacing, aspect ratios all hold
5. After Rclone sync:
   - Next page load shows the real image automatically
   - No code change required
```

This is the Ghost Operator philosophy applied to media: **the engine stays stateless, the browser handles reality.**
