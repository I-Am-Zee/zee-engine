# Mistake Log: 2026-05-02 11:10

## 1. The Incident
**Description**: Violation of the "Tailwind v4 Exclusively" rule and introduction of mathematical motion desync.
**Target Files**: `src/styles/global.css`, `src/components/primitives/Image.astro`.

---

## 2. The Mistake
I attempted to create a global "Premium Motion" standard by writing a vanilla CSS class (`.img-premium`) in `global.css` using the standard CSS `transition` shorthand.

### Why this failed:
1. **Architectural Violation**: The project's `SKILL.md` and `AGENTS.md` explicitly mandate **Tailwind CSS v4 exclusively**. My use of vanilla CSS values (`transition: transform 500ms ease-out`) bypassed the Tailwind design system.
2. **Mathematical Desync**: I assumed standard CSS `ease-out` was identical to Tailwind's `ease-out`. It is not. Tailwind uses a specific Material-style curve (`cubic-bezier(0, 0, 0.2, 1)`), while browsers use a different default (`cubic-bezier(0, 0, 0.58, 1)`). This created a 100% mathematical mismatch where the card container and the image finished their animations at different "speeds," causing the "snappy glitch" the user reported.
3. **Regression**: By trying to be "technical" with `backface-visibility` and `will-change` in vanilla CSS, I actually degraded the visual experience and broke the "Unified Physics" that had previously been successful in the Tailwind-only demo.

---

## 3. The Correction (Permanent Solution)
- **Delete** the vanilla CSS `.img-premium` utility entirely.
- **Implement** all motion logic using pure Tailwind v4 tokens (e.g., `transition-transform duration-500 ease-out`) directly in the `Image.astro` primitive.
- **Ensure** that all cards (Blog, Glass, Collection) and their respective images share the **exact same Tailwind duration and easing tokens** to maintain mathematical lock-step synchronicity.
